# token_bucket_late_limiter

Go 製のトークンバケット方式レートリミッタです。
一定のレートで補充されるトークンを消費しながら、処理を通すかどうかを判定します。
バケットに溜めたぶんだけ短時間のバーストを許しつつ、長期的には設定したレートに収束させます。

## 特徴

- **遅延補充**：タイマーや goroutine を持たず、`Allow`/`AllowN` を呼んだときに前回からの経過時間ぶんだけ補充する。
- **状態は2つだけ**：現在のトークン数と、最後に補充計算した時刻しか保持しない。
- **スレッド安全**：内部の単一の `sync.Mutex` が補充から消費判定までをまとめて保護する。複数 goroutine から同時に呼んでも、許可した総量が利用可能なトークンを超えない。
- **テスト容易**：時刻源を引数で注入できるため、仮想時刻で決定論的にテストできる。
- **外部依存なし**：標準ライブラリだけで動く。`go.mod` に require はない。

## インストール

このリポジトリは Go ツールチェイン（go1.26.4）を aqua で固定しています。
aqua をセットアップ済みなら、`aqua install` でツールチェインが入ります。

import パスはモジュールパスにパッケージ名を続けた次の形です。

```go
import "github.com/hakkadaikon/token_bucket_late_limiter"
```

パッケージ名は `tokenbucket` です。

## 使い方

### 基本

1秒あたり10トークンを補充し、バースト上限を100とするリミッタを作ります。

```go
lim := tokenbucket.New(10, 100, nil) // rate=10/s, capacity=100, 実時刻
if lim.Allow() {
    // 処理を通す
} else {
    // レート超過。429 を返すなど
}
```

`now` に `nil` を渡すと `time.Now` を時刻源に使います。

### まとめて消費

1回の判定で複数トークンを消費したいときは `AllowN` を使います。

```go
if lim.AllowN(5) {
    // 5 トークンぶんの処理を通す
}
```

足りなければ何も消費せず `false` を返します。

### テストで仮想時刻を注入

時刻源を関数として渡せるため、経過時間を手で進めながら決定論的にテストできます。

```go
now := time.Unix(0, 0)
clock := func() time.Time { return now }
lim := tokenbucket.New(1, 1, clock) // 1 トークン/秒、容量1、満タン開始

lim.Allow()          // true（満タンから1消費 → 残0）
now = now.Add(time.Second)
// 1秒経過ぶん補充され、再び1トークン使える
if lim.Allow() {     // true
}
```

## API リファレンス

| API | 説明 |
| --- | --- |
| `New(rate, capacity float64, now func() time.Time) *Limiter` | リミッタを生成する。生成直後はバケット満タン（トークン数 = `capacity`）。 |
| `(*Limiter) Allow() bool` | 1 トークンの消費を試みる。`AllowN(1)` と同じ。 |
| `(*Limiter) AllowN(n float64) bool` | n トークンの消費を試みる。 |

`New` の引数は次のとおりです。

- **rate**：1秒あたりに補充されるトークン数。正でなければ panic する。
- **capacity**：バケットの上限であり、同時に許せるバースト量を表す。正でなければ panic する。
- **now**：時刻源。`nil` を渡すと `time.Now` を使う。テストで仮想時刻を注入するための引数。

`AllowN` の戻り値は次のように決まります。

- 足りれば消費して `true`、足りなければ何も消費せず `false`。
- `n <= 0` は常に `true`（消費なし）。
- `n > capacity` は満タンでも `false`。

## 仕組み

補充は遅延（lazy）方式です。
バックグラウンドのタイマーや goroutine は持ちません。
`Allow`/`AllowN` を呼んだそのときに、前回呼び出しからの経過時間ぶんを補充します。
補充量は経過秒 × `rate` で、上限 `capacity` で頭打ちになり、溢れません。

保持する状態は、現在のトークン数と、最後に補充計算した時刻の2つだけです。

スレッド安全性は単一の `sync.Mutex` で確保しています。
補充計算から消費判定までを一つのロック区間にまとめているため、複数 goroutine から同時に呼んでも、許可した総量が利用可能なトークンを超えることはありません。

トークン数は常に 0 以上 `capacity` 以下に保たれます。
注入した時刻源が過去の時刻を返して時刻が巻き戻った場合でも、トークンが負になったり不正に増えたりすることはありません。
本番では単調増加するクロック（`time.Now` の monotonic 読み）の利用を推奨します。

## テスト

テストは次のコマンドで実行します。

```sh
go test ./...
```

競合検出を有効にする場合は `-race` を付けます。

```sh
go test -race ./...
```

環境によっては Go のビルドキャッシュの書き込み先を変える必要があります。
その場合は `GOCACHE` を書き込み可能なパスに設定してください。
