# レトロスペクティブ: lazy 補充トークンバケットを TLA+ と test-design で実装した記録

lazy 補充方式のトークンバケット rate limiter を、ループエンジニアリング(TLA+)とテスト設計カタログ(test-design)の両方を使って Go で実装した。
読み手は、次に同じ手法を使う開発者を想定する(将来の自分を含む)。
KPT の形を借りつつ、実際に何が起きてどの判断が効いたかを、管理番号のトレーサビリティとともに残す。

## 開発の流れ

開発は要件読解から始めた。
検証ゲートの3問(状態遷移・並行があるか、通常テストより強い保証が要る数学的性質があるか、それをどうテストリストに落とすか)は、要件ファイルにほぼ答えが書いてあった。
残量という状態に時間補充と並行 acquire が絡むので TLA+ は確定、lazy 補充は線形計算で Lean は任意、橋渡し先は TDD のテストリスト。
plan mode 相当の判断を、迷わず最初に下せた。

次に、loop-engineering と test-design を並行で起動した。
TLA+ の網羅検査(設計の正しさ)と振る舞い抽出(実装のテスト項目)は互いに独立なので、重い TLC 検査を modeler に委譲し、メイン文脈では test-design の振る舞い抽出を進めた。
片方が他方の出力を待つ依存が無かったため、並行が素直に効いた。

TLA+ 側は、自然言語要求を EARS の抽出台帳(S-001 から S-014)に落とすところから始めた。
TLC は float を扱えないので、tokens・capacity・rate・dt をすべて整数にスケーリングして離散化した(S-001 の離散化方針)。
そのモデルを TLC で網羅探索し、2600 状態を踏んで INV1 から INV4 まで反例ゼロを確認した。
ここで止めず、spec 自体を mutation testing にかけた(M1 から M7)。

test-design 側は、用語定義を先に埋めてから振る舞いを20項目(T-001 から T-060)抽出した。
0段のクローズ(未チェック0件、欠番チェック)を通したうえで、property テストを TLA+ の不変条件に1対1で橋渡しした(PROP↔INV)。

実装は coder agent による TDD で進めた。
台帳の T-ID を1項目ずつ Red から Green へ落とし、property based testing には標準ライブラリの testing/quick だけを使った。

最後に検証ゲートを通した。
agent の報告を鵜呑みにせず、自分で go vet と go test -race -count=5 を実行し、20件すべて PASS かつ flaky でないことを確認した。
変更は micro-commit で3コミットに分割した。

## Keep

検証ゲートを先に通したことで、実装の手戻りがゼロだった。
実装ファーストで書き始めず、設計層(TLA+)と検証層(テストリスト)を先に固めてからコードに入った。
「後から検証層を足すより最初に層を決める方が安い」を、そのまま実地でなぞれた。

TLA+ と test-design の並行起動も効いた。
設計の網羅検査と振る舞い抽出は独立な作業なので、modeler に重い検査を委ねるあいだ、メイン文脈は振る舞い抽出だけに専念できた。

mutation oracle が、検査そのものの有効性を保証した。
INV を書いて反例ゼロを確認しただけでは、その INV が実際に何かを捕まえている保証にはならない。
そこで spec をわざと壊し、各変異が正しい INV で KILLED されることを確かめた。

| 変異 | 壊した内容 | kill した INV |
|---|---|---|
| M1 | clamp(`Min(Capacity, ...)`)を除去 | Inv1 |
| M2 | Succeed ガードを `tokens >= 0` に緩和 | 過剰消費を Inv1/Inv3 が検出 |
| M3 | 消費量を1ずらす(保存則破壊) | Inv3 |
| M4 | Enter の lock ガードだけ除去(相互排他喪失) | MutexInv |
| M5 | refill を1多く(過剰補充) | Inv1 |
| M7 | granted の更新漏れ(保存則の granted 側破壊) | Inv4 |

真の安全性 survivor をゼロにすることを打ち切り基準にした。

この mutation を効かせる前提として、TypeOK から tokens の範囲を意図的に外した設計判断が地味に効いた。
tokens の範囲は TypeOK ではなく Inv1 の管轄に置いた。
そのため M1(clamp 除去)が TypeOK ではなく Inv1 で kill され、「Inv1 が実際に範囲を縛っている」ことまで示せた。
範囲を TypeOK に含めていたら、M1 は型エラーで落ち、Inv1 が効いているかどうかは確かめられなかった。

TLA+ の不変条件は、TDD のテストリストに1対1で橋渡しした。

| 設計(TLA+) | 実装テスト(test-design) | 内容 |
|---|---|---|
| INV1 | T-040 | 任意の操作列でも残量が範囲内(PBT) |
| INV3 | T-041 | 成功条件と消費量(PBT) |
| INV4 | T-050 | 並行 acquire の総許可量 |

設計と実装テストが、同じ性質を別レベルで二重に縛る形になった。

モデル検査は、実装の勘所を先に確定させた。
clamp は補充直後に置く(後段クランプ禁止)、last 更新は成否に依らず必ず行う、mutex は補充から消費までを1クリティカルセクションに含める。
これらは「書いてみて気づく」のではなく、モデルから先に出た。
mutex の分割が許されないことは、M4(Enter の lock ガード除去)が MutexInv 違反で kill された事実が根拠になっている。

検証ゲートでは、agent の報告を信用せず自分で再実行した。
go test -race -count=5 まで回し、flaky でないことを目で確認した。

clock の注入は最小の seam にとどめた。
既定は time.Now を使い、テスト時だけ `func() time.Time` を差し込めるようにした。
これで補充ちょうど1秒(T-013)と0.999秒(T-014)の境界を、決定論的に検証できた。

## Problem

mutation で survivor が1つ残った。
M6(Fail ガードの反転、足りているのに失敗する変異)である。
これは安全性 INV の管轄外だった。過小許可は安全側に倒れるので、安全性検査では原理的に kill できない。
見落とすと「足りているのに失敗する」退行を誰も捕まえないまま通り抜ける。
正常系の活性テスト T-060(供給十分なら成功する)を明示的に立てて、この穴を埋めた。
安全性だけを見ていると正常系の活性が盲点になる、という学びが残った。

float と TLC の不整合も引っかかった点である。
TLC は float を扱えないので整数スケーリングで離散化した。
モデル内の丸め誤差はこれでゼロになるが、実装側の float64 の丸め(tokens がわずかに capacity を超える、あるいは負になる経路)はモデルの管轄外に残る。
実装は math.Min によるクランプと property テストで範囲を締めたが、ここはモデル検査では届かない。
モデル検査は万能ではなく、離散化の境界に穴が残りうる。

活性(LIVE1)は到達性の確認にとどめ、完全には証明していない。
公平性条件下の temporal 検証は重く、要件でも任意だったので YAGNI で見送った。
完全にやるならコストが跳ねる、という判断ポイントとして記録しておく。

環境側の罠も2つあった。
1つは、go のビルドキャッシュ既定(~/.cache/go-build)が read-only だった点で、GOCACHE をリポジトリ内に向けて回避した。
aqua で go を入れても、実行環境のキャッシュ書き込み権限は別問題として残る。
もう1つは、aqua 本体が PATH になく、go バイナリの実体(.aqua/pkgs 配下の go/bin/go)を掘り当てる必要があった点である。
aqua-proxy は aqua 本体が PATH にあることを前提に動くが、それが満たされない環境だった。

## Try

mutation の survivor 分類を、最初からルール化する。
「safety で kill できない survivor は正常系か活性の対象」と判で押すように分類し、対応する正常系テストを必ず1本立てる。
今回は M6 から T-060 を後追いで足したが、次はこの分類を先回りで行う。

離散化の穴(float の丸め)を埋めるテストを、モデル化と同時にテストリストへ予約する。
「モデルは整数、実装は float」のギャップは毎回出るので、テンプレ化して T-ID を先に確保しておく。

環境セットアップ(GOCACHE、aqua の go の実体パス)を Makefile か簡単なスクリプトに固めるかは、要検討にとどめる。
1人開発で毎回手で通すだけなら過剰(YAGNI)。
チーム開発に渡す段階で入れればよい。

Lean を入れる損益分岐を意識しておく。
今回は lazy 補充が線形計算で自明であり、INV は TLA+ と property テストで二重に縛れたので、Lean はスキップして正解だった。
証明が要るのは、テストや網羅検査では到達できない無限領域の数学的性質を扱うときである。
この線引きを次回も持っておく。

## 全体所感

検証の三層(設計は TLA+、実装の証明は Lean、実装の駆動は TDD)を役割で使い分け、Lean を外す判断をしたのが良かった。
三層すべてを無理に使わず、性質の自明さに応じて層を落とせた。
設計を先に固めたことで、実装の手戻りがゼロだった。

手法の本質は、検証器を検証対象の上位に置くことにある。
INV を書く(検証対象)だけでなく、mutation で INV 自体を検証した(上位の検証器)。
この構図が、今回いちばん効いた一手だった。
